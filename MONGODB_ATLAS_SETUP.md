# MongoDB Atlas Setup — Step-by-Step for Illuminate Studios

The Illuminate Studios backend uses MongoDB for everything that isn't a photo: users, bookings, galleries metadata, documents, invoices, Luma chat sessions. Photo binaries go to Cloudflare R2 (see `CLOUDFLARE_R2_SETUP.md`).

MongoDB Atlas is MongoDB's managed cloud service. The free tier (M0) is plenty for a single-photographer studio's first year. You'll only need two env vars on Railway:

```
MONGO_URL=mongodb+srv://USER:PASS@cluster0.xxxxx.mongodb.net/?retryWrites=true&w=majority
DB_NAME=illuminate_studios
```

---

## Step 1 — Create an Atlas account

1. Go to https://www.mongodb.com/cloud/atlas/register.
2. Sign up with email + password or Google.
3. Answer the welcome questions (any answers fine; pick **"My application"**, language **Python**).

---

## Step 2 — Create a project

1. Top-left → **Projects** → **+ New Project**.
2. Project name: `Illuminate Studios`.
3. Click **Next**, skip the "Add members" step, click **Create Project**.

---

## Step 3 — Deploy a free cluster

1. From the project → **+ Create** (or "Build a Database").
2. **Choose a deployment type**: pick **M0 FREE**.
3. **Provider & region**: pick **AWS** and region **ap-southeast-2 (Sydney)** for best AU latency. (If Sydney isn't available on free tier, pick the closest — usually Singapore.)
4. **Cluster name**: leave as `Cluster0` (or rename if you prefer — note that name for later).
5. Click **Create Deployment**.
6. Cluster provisioning takes ~3–5 minutes. While it's working, Atlas will pop a "**Connect to your cluster**" wizard — don't close it.

---

## Step 4 — Create a database user

In the wizard (or via Database Access in the sidebar):

1. **Username**: `illuminate_app`
2. **Password**: click **Autogenerate Secure Password** → **copy and save it immediately**. You'll need it for the connection string.
3. **Built-in role**: **Read and write to any database** (for simplicity; can be scoped down to `illuminate_studios` only — see Step 8).
4. Click **Create Database User**.

---

## Step 5 — Allow network access

1. Sidebar → **Network Access** → **+ ADD IP ADDRESS**.
2. For Railway, the simplest option is to **allow all IPs** (because Railway services don't have a fixed outbound IP on the free tier):
   - Click **Allow access from anywhere** → IP address `0.0.0.0/0`.
   - Optional comment: `Railway production`.
3. Click **Confirm**.

> **Security note**: `0.0.0.0/0` is safe because authentication is enforced by the username/password — anyone hitting the endpoint without credentials gets rejected. If you want belt-and-braces, upgrade to Atlas M10 ($57/month) which supports VPC peering / PrivateLink with Railway.

---

## Step 6 — Get the connection string

1. Sidebar → **Database** → click your cluster → **Connect**.
2. Pick **Drivers** as the connection method.
3. Driver: **Python**, version **3.12 or later**.
4. **Copy** the connection string. It looks like:
   ```
   mongodb+srv://illuminate_app:<password>@cluster0.abcde.mongodb.net/?retryWrites=true&w=majority&appName=Cluster0
   ```
5. **Replace `<password>`** with the actual password from Step 4 (URL-encode it if it contains `@`, `:`, `/`, etc. — Atlas autogenerated passwords are safe).

That string is your `MONGO_URL` env var.

---

## Step 7 — Paste env vars into Railway

In your Railway **`illuminate-backend`** service → **Variables**:

```
MONGO_URL=mongodb+srv://illuminate_app:<password>@cluster0.xxxxx.mongodb.net/?retryWrites=true&w=majority
DB_NAME=illuminate_studios
```

Railway redeploys automatically. The backend startup logs should show:

```
INFO illuminate Seed complete
INFO:     Application startup complete.
```

(No errors. If you see `pymongo.errors.ServerSelectionTimeoutError`, double-check Network Access in Step 5.)

---

## Step 8 — Scope the database user (optional hardening)

The default user has "Read and write to any database" which works fine but is more permissive than needed.

1. Atlas → **Database Access** → click the `illuminate_app` user → **Edit**.
2. Under **Specific Privileges**, add:
   - **Database**: `illuminate_studios`
   - **Collection**: leave blank (= all collections in that db)
   - **Role**: `readWrite`
3. Remove the broad role (`atlasAdmin` or `Read and write to any database`).
4. **Update User**.

The app continues working but the user can only touch the `illuminate_studios` database.

---

## Step 9 — One-time index check

The backend creates indexes on startup automatically (see `seed.py → run_seed`). After first deploy, verify:

1. Atlas → your cluster → **Collections** → switch to the `illuminate_studios` database.
2. You should see the following collections (created lazily as data is written):
   - `users` (unique index on `email`)
   - `bookings` (index on `user_id`)
   - `galleries` (index on `client_user_id`)
   - `documents`, `invoices`, `luma_sessions`, `service_packages`, `service_addons`, `portfolio`, `file_blobs` (if not using R2), `counters`, `payment_transactions`

If a collection is missing, just exercise the corresponding feature (e.g. create a booking) and it'll appear.

---

## Step 10 — Backups (free tier limitations)

The M0 free tier **does not** include automated backups. For peace of mind on a production studio:

- **Option A — upgrade to M10** ($57/month) which gives you continuous backups + point-in-time restore.
- **Option B — manual `mongodump`** weekly via a Railway cron service (cheap workaround):
  ```bash
  mongodump --uri="$MONGO_URL" --out=/backups/$(date +%F)
  ```
  Then upload the dump folder to your R2 bucket as `backups/yyyy-mm-dd/`.

For the first 3–6 months of operation, manual weekly dumps are reasonable. As soon as your real client list grows, upgrade to M10.

## Cost sanity check (Feb 2026 pricing)

| Tier | Cost | Storage | Backups | When to use |
|---|---|---|---|---|
| **M0 Free** | $0 | 512 MB | No | First 6–12 months. Plenty for a single studio. |
| **M2** | ~$9/mo | 2 GB | No (snapshots only) | Once you exceed 512 MB. |
| **M10** | ~$57/mo | 10 GB | Yes (continuous PITR) | Once you have real revenue. |

A 12-month-old studio with ~200 bookings, ~50 contracts, ~300 invoices, ~100 Luma sessions, ~2000 photo metadata rows will use ~50 MB. You'll comfortably stay on M0 for the first year.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| `ServerSelectionTimeoutError` | Network Access doesn't allow Railway IPs | Set `0.0.0.0/0` in Network Access (Step 5) |
| `Authentication failed` | Wrong password in URL or password not URL-encoded | Recopy connection string, re-replace `<password>` |
| App starts but `/api/auth/login` returns 401 for admin | Seed hasn't run or wrong `DB_NAME` | Check backend logs for `Seed complete`. Verify `DB_NAME=illuminate_studios`. |
| Slow queries | Free-tier shared cluster | Upgrade to M10 or wait — usually only an issue at scale |
| "Cluster paused" message | M0 clusters pause after 60 days of inactivity | Click "Resume" in Atlas — your data is intact |
